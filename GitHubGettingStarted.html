Automating Tasks with GitHub Actions

Understanding GitHub Actions

  1. We can use GitHub for many parts of the SDLC
  2. Tasks can be automated
  3. Automate different tasks
  4. Event-driven, will trigger tasks
  5. Live in the repository
  6. Stored on workflows folder
  7. Actions are based on YAML syntax
  8. Part of GitHub
  9. Templates
  10. Available in free plan

Components of GitHub Actions

  1. Workflow
  2. Workflow contains jobs
  3. Event triggers a workflow
  4. Workflow contains one or more jobs
  5. Each job as steps and actions
  6. Executions of these Jobs are done on Runners
  7. Runners: servers that runs the automation code
  8. it will typically server hosted by GitHub or it can be your own
  9. Steps can be action or command line commands
  10. actions are the fineist building block of GitHub Actions they are the standalone commands
  11. we can create our own actions

Creating an action in YAML

  1. first line - name: first-github-action //name of the workflow
  2. second line - on: [push] //when to trigger the workflow here in this e.g. on push. on represents event
  3. third line - jobs: //all jobs contained within this workflow
  4. fouth line - check-bats-version: //check-bats-version is the name of the job
  5. fifth line - runs-on: ubuntu-latest //runner hosted by github
  6. sixth line - steps: //grouped by steps
  7. seventh line - -uses: actions/checkout@v2 // actions are the smallest tasks it will checkout on the server
  8. eigth line - - uses:actions/setup-node@v1 //setup node on the runner server
  9. ninth line - -run: npm install -g bats //install bats package on the server, bats package is used for testing the applications
  10. tenth line - -run: bats -v //command to get the bats version
